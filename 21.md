## Exercise 45

> 在下面这些引擎里，和 Room 最接近的概念分别是什么：Unity、GODOT、HaxeFlixel、Construct 2 和 Phaser。去翻一下它们的文档，试着找出来。顺便也看看这些对象通常有哪些方法，以及它们是怎样在不同“房间”之间切换的。

Room 在别的引擎里通常叫 Scene 或 State。名字不同，职责接近。下面是常见对应和切换方式，适合做类比；具体 API 名称会随版本变化，不要把它当成硬性接口。

Unity 最接近的是 Scene。常见生命周期是 Awake/Start/Update/OnEnable/OnDisable。切换通常是加载场景，单场景或叠加加载。

Godot 最接近的是 Scene（PackedScene 加 Node 树）。常见生命周期是 `_enter_tree`、`_ready`、`_process`、`_physics_process`、`_exit_tree`。切换可以用 `change_scene_to_file`，或实例化后 `add_child`。

HaxeFlixel 最接近的是 FlxState，必要时用 FlxSubState。常见生命周期是 create/update/draw/destroy。切换常见是 `FlxG.switchState`。

Construct 2 最接近的是 Layout（配 Event Sheet）。常见生命周期是 “On start of layout / On end of layout”。切换是 System 事件里的 “Go to Layout”。

Phaser 2.6.2 最接近的是 State。常见生命周期是 init/preload/create/update/render/shutdown。切换常用 `game.state.start`。

## Exercise 46

> 随便挑两个单机游戏，像我分析 Nuclear Throne 和 Isaac 那样，把它们拆解成一组 Room。尽量现实一点地思考，认真判断哪些界面或流程真的该独立成 Room，哪些不该。再进一步说明：在什么时机下会调用 `addRoom`，又会在什么时机下调用 `gotoRoom`。

只把逻辑完全不同的界面拆成 Room。像暂停、提示框这类覆盖层，用同一个 Room 内部状态就够了。`addRoom` 只在需要“回头继续”的场景才用，普通切换直接 `gotoRoom`。

例 1：Celeste（平台跳跃）。可以拆成 Boot -> MainMenu -> FileSelect -> Level -> ChapterEnd -> MainMenu。暂停、辅助选项、地图只是覆盖层，留在 Level 里。`addRoom` 只在你希望“回到刚才那一关还能保留状态”时用，例如章节内回头取道具；普通选关、过场、死亡重试则直接 `gotoRoom`。

例 2：Slay the Spire（卡牌肉鸽）。可以拆成 MainMenu -> RunSetup -> Map -> Battle -> Reward -> Map -> Shop/Event -> GameOver。牌组、遗物、设置页本质是覆盖层，不必拆 Room。`addRoom` 只有在你想回看上一个战斗或商店并保留当时状态时才需要，否则节点点击进入战斗/事件、战斗结束进奖励、奖励结束回地图都用 `gotoRoom` 即可。

## Exercise 47

> 从整体上说，Lua 的垃圾回收器是怎样工作的？Lua 里为什么会出现内存泄漏？又有哪些办法能防止这类问题，或者至少在它发生时把它检测出来？

Lua 的 GC 是追踪式的标记-清扫，会在运行中分批做。它先从根对象出发标记可达对象，再清掉不可达对象。内存泄漏多半不是“回收器失效”，而是引用没断：全局表、缓存表、事件监听、闭包捕获、Timer 队列、单例里存了对象列表等。

预防的思路很简单：对象不用了就断引用；缓存要可控，必要时用弱表；事件监听与定时器要成对注销；大表别长期挂在全局。检测层面，多看内存曲线与对象数量变化，跑一段时间观察是否持续上升；再用分区开关法缩小范围，逐段禁用功能看内存是否回落。

## Exercise 48

> 创建一个带有 `Area` 的 `Stage` Room。然后再创建一个继承自 `GameObject` 的 `Circle` 对象，并让 `Stage` 每隔 2 秒在随机位置生成一个这样的实例。每个 `Circle` 实例都应该在 2 到 4 秒之间的随机时间后自我销毁。

Area 管对象列表，Stage 只负责“何时生成”。Circle 继承 game_object，生命周期用倒计时最直观。如果项目里已有 timer，可以把寿命改成 `timer:after(t, kill)`，逻辑更清爽。

stage = object:extend()

function stage:new()
    self.area = area(self)
    self.spawn_timer = 0
end

function stage:update(dt)
    self.spawn_timer = self.spawn_timer + dt
    if self.spawn_timer >= 2 then
        self.spawn_timer = self.spawn_timer - 2
        local x = love.math.random(40, 760)

        self.area:add_game_object("circle", x, y, { life = life })

end

```lua [objects/game_object]
game_object = object:extend()

function game_object:new(area, x, y, opts)
    local opts = opts or {}
    if opts then for k, v in pairs(opts) do self[k] = v end end
    self.area = area
    self.x, self.y = x, y
    self.dead = false
end
```

```lua [objects/circle]
circle = game_object:extend()

function circle:new(area, x, y, opts)
    circle.super.new(self, area, x, y, opts)
    self.r = self.r or 10
    self.life = self.life or 2
end

function circle:update(dt)
    self.life = self.life - dt
    if self.life <= 0 then self.dead = true end
end

function circle:draw()
    love.graphics.circle("line", self.x, self.y, self.r)
end
```

```lua [objects/area]
area = object:extend()

function area:new(room)
    self.room = room
    self.game_objects = {}
end

function area:update(dt)
    for i = #self.game_objects, 1, -1 do
        local obj = self.game_objects[i]
        obj:update(dt)
        if obj.dead then table.remove(self.game_objects, i) end
    end
end

function area:draw()
    for _, obj in ipairs(self.game_objects) do obj:draw() end
end

function area:add_game_object(class_name, x, y, opts)
    local obj = _G[class_name](self, x or 0, y or 0, opts or {})
    table.insert(self.game_objects, obj)
    return obj
end
```

## Exercise 49

不依赖 area，也不继承 game_object。Room 自己维护列表和删除逻辑。适合非常小的原型；对象多时就会变成重复劳动。

stage = object:extend()

function stage:new()
    self.spawn_timer = 0
    self.circles = {}
end

function stage:update(dt)
    self.spawn_timer = self.spawn_timer + dt

        self.spawn_timer = self.spawn_timer - 2

        local y = love.math.random(40, 560)

        local c = self.circles[i]
        c:update(dt)
        if c.dead then table.remove(self.circles, i) end

    for _, c in ipairs(self.circles) do c:draw() end
end

## Exercise 50

> 第 1 题的解法里引入过一个 `random` 函数。请扩展这个函数，让它在只传一个参数时，也能正常工作，并返回一个介于 0 和该参数之间的随机实数。另外，再扩展一下，让 `min` 和 `max` 可以反着传，也就是第一个值允许比第二个值更大。

单参时视为 `0..max`，并允许 `min > max` 的情况先交换。这样调用更自由，但会让“传参错误”更难被发现，调试时要有心理准备。

```lua
function random(min, max)
    if max == nil then
        max = min
        min = 0
    end

    if min > max then
        min, max = max, min
    end

    if min == max then return min end
    return love.math.random() * (max - min) + min
end
```

## Exercise 51

> `addGameObject` 函数里的 `local opts = opts or {}` 这一句，作用到底是什么？

`opts or {}` 的目的是保证 `opts` 始终是表，避免 `pairs(nil)` 报错，并让“可选参数”调用更顺滑。它不适合需要区分“没传”和“传了空表”的场景；如果你需要这种区分，就要显式检查 `opts == nil`。
